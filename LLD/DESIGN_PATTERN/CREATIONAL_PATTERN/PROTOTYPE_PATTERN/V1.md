# 1️⃣ First: what is ACTUALLY happening in your code?

### This line is the key:

```java
return supplier.get().clone();
```

### What does `supplier.get()` do?

```java
WelcomeMailTemplate::new
```

This means:

➡ **Every call to `supplier.get()` creates a BRAND NEW object**

So the flow is:

```
supplier.get()        -> new WelcomeMailTemplate()
.clone()              -> clone of that brand-new object
```

### 🔴 Conclusion (very important)

❌ You are **NOT** cloning a shared prototype
❌ You are creating a **new object and then cloning it immediately**
❌ The clone is **wasted effort**

So effectively, your code behaves like:

```java
return new WelcomeMailTemplate();
```

➡ **This is NOT a real Prototype Pattern implementation**

---

# 2️⃣ What Prototype Pattern REALLY means (Principal-level clarity)

### Definition (interview-grade)

> Prototype Pattern means **cloning a pre-configured instance**, not creating a new one every time.

### Correct mental model

```
Registry
  |
  | holds ONE prototype instance
  |
  v
Prototype Object
  |
  | clone()
  v
New customized copy
```

### Why Prototype exists

* Object creation is **expensive**
* Configuration is **complex**
* We want **fast copies**, not reconstruction

### ❌ Problem 1: Misuse of Supplier

Supplier implies **Factory**, not **Prototype**.

### ❌ Problem 2: Cloneable is fragile

`Cloneable`:

* breaks constructors
* shallow copy by default
* error-prone

### ❌ Problem 3: Registry is static & rigid

* Hard to test
* Hard to extend

# 4️⃣ Correct way to implement Prototype Pattern (Production-grade)

## ✅ Key decisions (and reasons)

### ✔ Store actual prototype instances (NOT suppliers)

Reason: Prototype pattern clones **existing objects**

### ✔ Avoid `Cloneable`

Use **copy constructor** or **explicit clone method**

### ✔ Registry holds immutable base templates

Safer, clearer, testable

# 5️⃣ Refactored — REAL Prototype Pattern

---

## 🔹 Step 1: Prototype interface (no Cloneable)

```java
interface EmailTemplate {
    EmailTemplate copy();
    void send(String to);
    void setContent(String content);
}
```

---

## 🔹 Step 2: Concrete prototype

```java
class WelcomeMailTemplate implements EmailTemplate {

    private String subject;
    private String content;

    public WelcomeMailTemplate(String subject, String content) {
        this.subject = subject;
        this.content = content;
    }

    // Copy constructor
    private WelcomeMailTemplate(WelcomeMailTemplate other) {
        this.subject = other.subject;
        this.content = other.content;
    }

    @Override
    public EmailTemplate copy() {
        return new WelcomeMailTemplate(this);
    }

    @Override
    public void setContent(String content) {
        this.content = content;
    }

    @Override
    public void send(String to) {
        System.out.println(
            "Sending to " + to + " [" + subject + "] " + content
        );
    }
}
```

---

## 🔹 Step 3: Proper Prototype Registry

```java
class EmailTemplateRegistry {

    private static final Map<EmailTemplateType, EmailTemplate> templates = new HashMap<>();

    static {
        templates.put(
            EmailTemplateType.WELCOME,
            new WelcomeMailTemplate(
                "Welcome!",
                "Default welcome content"
            )
        );
    }

    public static EmailTemplate getTemplate(EmailTemplateType type) {
        EmailTemplate prototype = templates.get(type);
        if (prototype == null) {
            throw new IllegalArgumentException("Unsupported template type");
        }
        return prototype.copy(); // REAL prototype cloning
    }
}
```

---

## 🔹 Step 4: Usage

```java
public class Main {
    public static void main(String[] args) {

        EmailTemplate mail1 =
            EmailTemplateRegistry.getTemplate(EmailTemplateType.WELCOME);
        mail1.setContent("Version 1");
        mail1.send("Bishwa");

        EmailTemplate mail2 =
            EmailTemplateRegistry.getTemplate(EmailTemplateType.WELCOME);
        mail2.setContent("Version 2");
        mail2.send("Rahul");
    }
}
```

# 6️⃣ Why THIS is correct Prototype Pattern

### ✔ Single prototype instance stored

### ✔ No unnecessary object creation

### ✔ Cloning is intentional and meaningful

### ✔ Safe copy semantics

### ✔ Easy to extend

Add new template:

```java
templates.put(RESET_PASSWORD, new ResetPasswordTemplate(...));
```

No existing code changes → **Open/Closed**

# 7️⃣ When SHOULD you use Supplier + Factory instead?

If:

* Objects are cheap
* No shared base configuration
* No cloning semantics

Then **Factory Pattern alone is better**

👉 Prototype is for **“clone instead of build”**
